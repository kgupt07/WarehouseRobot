1. Project Summary & Scope

One-sentence version

In simulation, build a small wheeled robot that knows where it is in a map, drives to chosen spots on its own, spots a target with a camera, reports success, and drives back home.

What the robot does (end state)

  1. Localizes — knows where it is on a 2D map (AMCL).
  2. Navigates — drives to a pose you give it (Nav2).
  3. Perceives — camera finds a target (ArUco marker recommended).
  4. Reports — publishes a message (e.g. target_found).
  5. Returns home — Nav2 drives back to the home pose.

Mission story

“Go to the search area, find the package (marker), report success, return to base.”

In scope

  • Localization on a 2D occupancy grid map
  • Autonomous navigation via Nav2
  • Camera-based target detection (start with ArUco)
  • Mission state machine (Python node you write)
  • Map: pre-made first; building map yourself (SLAM) as milestone or stretch

Out of scope (month 1)

  • Robot arm / tapping or touching the target (stock TurtleBot 3 has no manipulator)
  • Training a large deep-learning model from scratch
  • Multi-robot coordination
  • Custom robot URDF from scratch in week 1

Time budget

ItemValue
Total hours~90–100
Daily focus~3–3.5 hours
Duration~4 weeks + Week 0

Priority rule: If time runs short, protect navigation + return home before fancy perception.

2. Project Description — Measurable Learning Levels

Learning is the underlying goal, but these checkpoints make progress tangible.

Level 1 — “I can run and observe the stack” (Week 1)

Done when:

  • Robot spawns in simulation
  • You can drive it with the keyboard (teleop)
  • You can see wheel odometry (/odom), laser scan (/scan), and velocity commands (/cmd_vel)
  • You can draw which nodes talk to which topics

You can explain: What is a node? What is a topic? What does the robot publish vs. what you send to move it?

Level 2 — “I understand space and mapping” (Week 2)

Done when:

  • You can explain the TF tree (coordinate frames: map, odom, base_link, sensor frames)
  • You have a map file loaded in RViz (built via SLAM or pre-made)
  • Laser scan aligns with walls on the map

Terms: TF = system tracking where each part of the robot is relative to the world. SLAM = building a map while estimating where you are.

You can explain: Mapping vs. localization? Why more than one coordinate frame?

Level 3 — “I can navigate on purpose” (Week 3)

Done when:

  • Robot localizes on saved map (AMCL)
  • You send a goal pose; Nav2 drives there without keyboard
  • Robot completes 3 goals in a row (A → B → C) with at most minor recovery events

Terms: AMCL = localization that matches laser scans to the map. Nav2 = standard ROS 2 navigation (planner + obstacle avoidance + controller).

You can explain: What Nav2 does vs. what your mission code does?

Level 4 — “Perception hooks into mission logic” (Week 4)

Done when:

  • Camera detects target (ArUco or colored object)
  • Mission node runs: HOME → GO_TO_SEARCH → DETECT → REPORT → GO_HOME → DONE
  • On detection, robot publishes target_found and returns home via Nav2

Stretch: Navigate closer to target before returning. No tapping on stock TurtleBot 3.

3. Reflection Answers & Plan Adjustments

Your stated priorities

  • Learning over demo polish — correct for month 1; use checkpoints above to avoid endless tutorial hopping.
  • Mapping / localization / navigation understanding — your mental model is sound; refinements below.
  • Mission: Find target and report (not tap). Signal/message is the right interaction for TurtleBot 3.
  • Environment: Windows 11 + WSL 2 + Ubuntu 22.04 (fresh install). Week 0 validates RViz/Gazebo before core content.

Mapping vs localization vs navigation (refined)

LayerWhat it does
MappingBuild a representation of the environment (usually 2D occupancy grid from lidar). Often done once, then map is saved and reused.
LocalizationEstimate where the robot is on that map (pose: x, y, yaw). AMCL compares live lidar to the map. Separate from mapping — you can localize on a map you did not build.
NavigationGiven a goal pose and current pose, compute a collision-free path and drive there. Nav2 handles this. Navigation does not decide which goal — mission logic does.
Mission logicYour code: “Go to search area, look for target, report, go home.”

Why “tap the target” is out of scope

Standard TurtleBot 3 (Burger/Waffle) has differential drive, lidar, optional camera — no arm or gripper. Tapping requires a manipulator (different project). Alternatives: publish target_found topic; optionally navigate closer to detected pose.

WSL 2 / RViz note

Windows 11 WSLg should show Linux GUI apps without extra X server. Week 0 proves rviz2 and simulation before SLAM/Nav2. If graphics stay painful, consider dual-boot Ubuntu.

4. System Architecture

┌─────────────────────────────────────────────┐
│  Mission node (YOU write this)              │
│  IDLE → GO_TO_SEARCH → DETECT → REPORT →    │
│         GO_HOME → DONE                        │
└──────────────┬──────────────────────────────┘
               │ goals + reads status
┌──────────────▼──────────────────────────────┐
│  Nav2 (navigation)                          │
│  Plans path, avoids obstacles, drives       │
└──────────────┬──────────────────────────────┘
               │ needs pose
┌──────────────▼──────────────────────────────┐
│  AMCL (localization)                        │
│  "Where am I on the map?"                   │
└──────────────┬──────────────────────────────┘
               │ uses
┌──────────────▼──────────────────────────────┐
│  Map (2D occupancy grid)                    │
│  Pre-made OR built with SLAM                │
└──────────────┬──────────────────────────────┘
               │ simulated sensors
┌──────────────▼──────────────────────────────┐
│  Gazebo + TurtleBot 3 + lidar + camera    │
└──────────────┬──────────────────────────────┘
               │
┌──────────────▼──────────────────────────────┐
│  ROS 2 (nodes, topics, actions, TF)       │
└─────────────────────────────────────────────┘

Key terms

  • Node — one running program in ROS (navigation, camera, your mission code).
  • Topic — named message stream (e.g. /scan, /camera/image_raw).
  • Action — long-running task with feedback; Nav2 uses NavigateToPose.
  • TF — transforms between coordinate frames (map, odom, base_link, camera).
  • URDF — robot description file (sizes, wheel placement, sensors).
  • Pose — position + heading: x, y, yaw.

5. Mission Flow

[START at HOME]
     │
     ▼
[GO_TO_SEARCH]  ──Nav2──►  drive to search pose (given)
     │
     ▼
[DETECT]        ──camera──►  look for target (ArUco)
     │
     ▼
[REPORT]        ──topic──►   publish target_found + details
     │
     ▼
[GO_HOME]       ──Nav2──►  drive back to home pose
     │
     ▼
[DONE]

Mission node state spec (Week 4)

StateAction
IDLEWait for start (service, topic, or timer)
GO_TO_SEARCHCall Nav2 to search pose; on success → DETECT
DETECTWait N seconds; if detection → REPORT; else retry or fail gracefully
REPORTPublish report message → GO_HOME
GO_HOMENav2 to home pose → DONE
DONEPublish mission complete

Record home pose and search pose in docs/poses.md once the map works.

9. Week 2 — Space, TF, and Maps (~22–26 hours)

Objective

Understand where the robot is in space; load or build a map.

Concepts

TF/tf2, frames (map, odom, base_link, base_scan, camera), SLAM Toolbox, map save/load, AMCL preview.

Deliverables

  • G2: Map file (.yaml + .pgm) — SLAM or pre-made
  • TF diagram: map → odom → base_link → sensors
  • RViz: map + laser aligned
  • Write-up: mapping vs localization in your words

Two paths

Path A (faster): Use default TurtleBot 3 navigation demo map; focus on TF + RViz. Defer self-built map to stretch.

Path B (full): Teleop + SLAM; save map with map_saver; reload in new session.

Tasks

  1. ros2 run tf2_tools view_frames
  2. RViz: Fixed Frame = map; add Map, LaserScan, TF
  3. SLAM + teleop OR load existing map

Common mistakes

  • Saving map before exploring full space

11. Week 4 — Perception, Report, Full Mission (~20–24 hours)

Objective

Find target → report → return home under mission control.

Concepts

Image topics, ArUco detection, custom/std messages, finite state machine, Nav2 + perception integration.

Deliverables

  • G4: Full state machine runs end-to-end
  • Publish /mission/target_found (or similar) on detection
  • Return home after report
  • Complete docs/system_graph.md
  • Explain stack top-to-bottom (~5 min out loud)

Perception order (easiest first)

  1. ArUco marker in Gazebo (recommended)
  2. Colored blob (OpenCV)
  3. Small ML detector (stretch only)

Tasks

  1. View /camera/image_raw (rqt_image_view or RViz)
  2. Detection node: image in → target_detected out
  3. Place marker in Gazebo at search area
  4. Mission FSM with timeouts in DETECT
  5. Log to /mission/status
  6. End-to-end test 3×; document failure modes

Common mistakes

  • Blocking node in a tight loop (use timers / async callbacks)
  • No timeout in DETECT
  • Changing too many Nav2 YAML params at once

Optional challenges

  • Build own map with SLAM; run full mission
  • Navigate closer using estimated object pose
  • Rosbag replay for perception debugging

12. Definition of “Project Complete”

  1. Localization: AMCL on 2D map; laser aligns with map in RViz.
  2. Navigation: Given target pose, Nav2 drives there reliably.
  3. Mission: Search → detect → report on topic → home.
  4. Understanding: System graph + explain each layer without notes.
  5. Mapping (stretch): You built and saved the map yourself.
